# Plan: Undo in Toast After Deleting Task & List

Generated by scheduled task on 2026-06-10. Covers webapp (desktop + mobile web), native Android, native iOS, scheduled tasks, floater tasks, lists, and floater lists.

## TL;DR

Undo-in-toast is **already implemented end-to-end on all three platforms** for tasks (scheduled + floater + recurring instances), lists, and floater lists. The analysis found **one real gap**: the web sidebar list delete (`ListSidebarSection.tsx`) — single and bulk — fires the DELETE immediately and shows a plain "List deleted" toast with **no Undo**. Everything else is covered. The bulk of this plan is therefore: (1) fix that gap, (2) a verification checklist, (3) optional consistency improvements.

## How undo works today (shared pattern)

All platforms use the same client-side **delayed-commit** pattern — there is no backend soft-delete or restore endpoint, and none is needed:

1. **Stage**: item is removed from local cache/query cache only (UI updates instantly). No DELETE request sent.
2. **Toast with Undo**: shown for ~5s (web) / 8s (Android, iOS).
3. **Undo tapped** → staged state restored (refetch on web; staged-record restore on mobile). Server never saw the delete.
4. **Toast expires/dismissed** → real DELETE fires (web) or DELETE pending-mutation is queued and synced (Android/iOS). Race-guarded so commit/undo runs exactly once.

## Work item 1 (the actual gap): undo for sidebar list deletes on web

**File**: `tday-web/src/components/Sidebar/List/ListSidebarSection.tsx` (used by `SidebarContainer.tsx` and `MoreNavigationSheet.tsx`, so it's both the desktop sidebar and the mobile "More" sheet).

Currently `deleteListMutation` and `bulkDeleteListsMutation` call `deleteLists()` (DELETE `/api/list`) immediately in `mutationFn`, then toast without an action. Convert both to the staged pattern already used in `ListFormSheet.tsx`:

1. Extract/reuse the staged-list-delete logic from `ListFormSheet.tsx` (`handleDeleteList`) into a shared hook, e.g. `src/hooks/use-undoable-list-delete.ts`, built on `useUndoableDelete`. It should:
   - Prune `["listMetaData"]` (and `["list", id]`) caches for the requested IDs.
   - Navigate to `/app/tday` if the active list is among them (current `handleDeletedListsSuccess` behavior) — navigate at stage time, like `ListFormSheet` does.
   - `commit`: call `deleteLists(ids)` (keeps existing single/bulk request shape and `deletedIds` response handling).
   - `undo`: `invalidateQueries(["listMetaData"])` (+ `["list"]`, `["todo"]`) to refetch — server never saw the delete, so refetch restores everything including cascaded tasks.
2. Use it in both mutations. Toast copy: `appDict("listDeleted")` for single; for bulk add a pluralized key, e.g. `"listsDeleted": "{{count}} lists deleted"`, to all 10 locale files in `tday-web/messages/`. Action label: existing `t("undo")`.
3. Close the dialogs and clear selection at stage time (current `onSuccess` behavior moves to stage).
4. Edge cases:
   - Don't show "Undo" on error toasts (commit failure should surface the existing destructive toast and invalidate queries).
   - Multiple rapid deletes: `useUndoableDelete` instances are per-toast; verify two overlapping list deletes each commit/undo independently (Sonner supports stacked toasts).
   - Bulk delete navigating away from an active list that's in the selection.

Estimated scope: 1 new hook, edits to 1 component, 10 locale files, plus tests.

## Work item 2: verification checklist (all platforms)

Manual/automated checks to confirm the existing implementations actually behave correctly:

- **Web** (`tday-web/tests/`): unit tests for `use-undoable-delete` double-commit guard if not present; Playwright/RTL checks that DELETE is not sent when Undo is clicked (assert no network call), and is sent exactly once after toast auto-close. Cover: task, calendar task, recurring instance, floater, list (form sheet + sidebar single + sidebar bulk), floater list.
- **Android**: verify recurring-instance delete from calendar restores the correct instance on undo (`instanceDateEpochMs` match); verify undo after navigating away from a deleted list (coordinator's MainScope claim); verify offline: delete → undo with airplane mode on (no pending mutation should remain); delete → commit offline → mutation queued and synced later.
- **iOS**: same matrix as Android; additionally verify the 8.5s commit vs 8s toast timing means the Undo button is always actionable while visible, and that killing the app mid-window leaves the item intact after relaunch (commit never ran — confirm this self-healing is the intended behavior on iOS as it is on Android).
- **Cross-platform copy consistency**: "Task deleted" / "List deleted" / "Floater deleted" — Android uses "Floater deleted" while web uses `taskDeleted` for floaters; decide on one term.

## Work item 3 (optional, consistency/polish)

- **Align undo window**: web ~5s vs Android/iOS 8.5s. Consider 8s web toast duration for delete toasts to match mobile.
- **Bulk list delete on mobile apps**: Android/iOS have no sidebar bulk delete; if added later, reuse the stage/commit repo methods which already accept lists.
- **Backend (no change recommended)**: endpoints are hard-delete (`DELETE /api/todo`, `/api/todo/instance`, `/api/list`, `/api/floater`, `/api/floaterList`). The client-side delayed-commit makes a server-side restore endpoint unnecessary. Only revisit if a longer undo window (e.g., minutes) or cross-device undo is ever wanted — that would require soft-delete (`deletedAt`) + restore endpoints and is explicitly out of scope here.

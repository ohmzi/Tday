This note defines the current recurring-task model in plain language. The parent Todo row is the recurrence definition, while TodoInstance rows are exceptional edits layered on top of the generated schedule.

The goal is predictable behavior across web, Android, iOS, and the Ktor backend: the recurrence itself stays canonical, and per-occurrence changes never mutate the underlying rule for the whole series.

1. Canonical model

  • Todo stores the series definition: title, description, priority, dtstart, due, durationMinutes, timezone, rrule, and exdates.
  • Generated recurring occurrences are virtual. They are produced at runtime from the canonical parent plus its exclusions.
  • TodoInstance stores a single override: edited title, description, priority, or shifted timing for one occurrence only.

2. Identity rules

Every generated occurrence needs a stable identity so clients can complete, move, reorder, and render it consistently. The important fields are:

  • id: the parent todo id.
  • instanceDate: the original occurrence being referenced.
  • recurId: the serialized form of that anchored occurrence in override rows. 
Todo = recurrence definition
TodoInstance = exception row
instanceDate = canonical occurrence anchor
recurId = string form of that anchor

3. Behavioral rules

  • Editing one occurrence creates or updates an exception row.
  • Editing the series updates the parent todo and leaves one-off exceptions as explicit overrides.
  • Completing a recurring occurrence records completion against the anchored occurrence instead of marking the whole parent series complete.
  • Deleting one occurrence should behave like adding an exclusion, not like deleting the recurrence definition itself.

4. Practical guidance

The Kotlin backend should stay authoritative for RRULE expansion, exception merging, and timeline materialization. Shared KMP code can own DTOs, validation, and safe helper logic, but the final expansion logic should run in one place so every client sees the same series shape.

When in doubt, preserve the parent recurrence as source of truth and model single-occurrence edits as explicit exceptions rather than series mutations.
